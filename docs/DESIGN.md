# Vibecraft Design Principles

Guidelines for creating a fluid, alive visualization.

## Core Philosophy

**The village should feel inhabited, not animated.** Characters move with purpose, sounds confirm actions without demanding attention, and every Claude tool call maps to a legible place in a world the user can understand at a glance.

## Village Layout

The village has 9 fixed locations, each mapping to one or more Claude tools:

| Location | Tool(s) | Character goes here when... |
|----------|---------|----------------------------|
| Library | Read | Claude reads a file |
| Writer's Cottage | Write, NotebookEdit | Claude creates a new file |
| Workshop | Edit | Claude edits an existing file |
| Terminal Tower | Bash | Claude runs a command |
| Observatory | Grep, Glob | Claude searches for patterns |
| Signal Tower | WebFetch, WebSearch | Claude fetches from the web |
| Town Hall | Task | Claude spawns a subagent |
| Notice Board | TodoWrite | Claude updates its task list |
| Village Square | Idle / default | Claude is between tasks |

The layout is fixed — each scenario's buildings are themed to match the scenario's aesthetic, but the 9 locations and their tool mappings never change.

## Character Design

**One character per session.** Each managed Claude session gets its own sprite that moves independently. Multiple Claude instances share the same village, spreading horizontally when they occupy the same location so you can always distinguish them.

Character states:
- **idle** — standing in the village square, nothing active
- **walking** — CSS transition from one building to another
- **working** — arrived at destination, tool is running
- **thinking** — tool completed, Claude is processing the result
- **finished** — session completed (`stop` event received)

Each state has its own sprite image. Scenarios define 7 character archetypes; sessions cycle through them so every instance looks distinct.

## Scenario System

A scenario is a complete visual theme: background image, 9 building exteriors, and 7 characters with 5 state images each. Scenarios are generated by OpenAI + Stable Diffusion and stored in `public/scenarios.json`.

**Design constraint:** scenarios are visual skins, not structural changes. The 9 locations, their functions, and the tool mappings are fixed. A scenario changes *what it looks like*, never *what it means*.

## Animation Timing

- **Character movement** — CSS `transition` on `left`/`top`, 600–800ms ease-in-out
- **UI transitions** — 100–200ms (panel open/close, state changes)
- **Attention pulses** — 1–2s loop when a session needs input

**Principle:** Movement should be fast enough to feel responsive, slow enough to read.

## Sound Design

- **Tool sounds** — each tool has a distinct synthesized signature (see [SOUND.md](./SOUND.md))
- **State sounds** — completion, error, subagent spawn/despawn
- **Non-intrusive** — sounds inform without demanding attention

**Principle:** Sound confirms without interrupting flow.

## Feedback Layers

Good interactions have multiple simultaneous feedback layers:

1. **Immediate visual** — character begins walking (location highlighted)
2. **Audio** — tool sound plays as the character departs
3. **State label** — activity feed entry appears
4. **Arrival** — character reaches building, state switches to `working`

## Multi-Session Legibility

When multiple sessions are active:
- Each session's character has a distinct sprite (by `agentIndex`)
- Characters at the same location spread horizontally (56px × scale per character)
- The activity feed is per-session — click a session to scope the feed
- Session status indicators (idle/working/offline) appear in the session panel

## Attention to Detail

Small things that matter:
- Characters don't snap — they walk across the village
- Sessions that need input pulse visually in the session panel
- The activity feed shows the tool name and duration, not just "tool used"
- Subagents appear as small secondary sprites at Town Hall

**Principle:** Polish compounds. 10 small legibility improvements > 1 flashy feature.
